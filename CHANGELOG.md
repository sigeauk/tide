## [4.1.15] - 2026-05-07

### Fixed

- **TCP connection-pool exhaustion during Elastic sync caused by `Connection: close` header (added in 4.1.14) colliding with the 20-worker mapping fan-out (`app/elastic_helper.py:fetch_detection_rules`).** The defensive `Connection: close` header shipped in 4.1.14 (to survive proxies that drop idle keep-alives) forced a fresh TCP socket per request. Combined with the existing `ThreadPoolExecutor(max_workers=20)` used downstream for parallel index-mapping fetches and the urllib3 default `HTTPAdapter(pool_connections=10, pool_maxsize=10)`, the session's pool saturated within seconds — manifesting as `urllib3.connectionpool: Connection pool is full` warnings, evicted in-flight connections, and SSL `Max retries exceeded with url: ... (Caused by ResponseError('too many redirects'))` storms against Kibana proxies. Removed `Connection: close` from the headers dict and mounted an explicit `requests.adapters.HTTPAdapter(pool_connections=25, pool_maxsize=25, max_retries=3)` on both `http://` and `https://` immediately after `session = requests.Session()`. Pool now absorbs the 20-thread parallelism with headroom and retries transient proxy hiccups without surfacing as sync failures. The standalone `_make_session(api_key)` helper (test-connection / promotion paths) was deliberately left untouched — it is single-threaded and the sized adapter is only required on the parallel-fetch session.

- **`scripts/diag_sync.py` Section 3 ("Live Kibana auth check") probed the bare `/api/detection_engine/rules/_find` endpoint, bypassing the `/s/default/` space prefix that the production sync path now requires.** Reverse-proxied Kibana ingresses (the standardised TIDE deployment shape post-4.1.14) route on `/s/<space>/...` and respond 404 / 302 to the bare path, which made Section 3 falsely report a connection failure on stacks where sync itself worked correctly — i.e. the diagnostic was lying. Changed the probe URL to `f"{url}/s/default/api/detection_engine/rules/_find?per_page=1"` so Section 3 mirrors the URL shape used by the live sync code path. The sibling per-`(siem,space)` probe in Section 4 was already correct and was not modified.

- **`scripts/diag_sync.py` had a duplicate "9." section header — both "9. Dry Run Sync URL Construction" and "9. Live sync trace (per-client, read-only)" printed under the same number, making operator triage of long diagnostic dumps ambiguous.** When Section 9 (dry-run URL construction) was inserted in 4.1.14, only the file-level docstring section index was updated; the in-function `_line(...)` call inside `_check_live_sync_trace` was missed. Renumbered the live-sync-trace header to `_line("10. Live sync trace (per-client, read-only)")` to match the docstring index and restore unambiguous section numbering.

## [4.1.14] - 2026-05-07

### Fixed

- **`scripts/diag_sync.py` Section 3 ("Live Kibana auth check") probed the bare `/api/detection_engine/rules/_find` endpoint, bypassing the `/s/default/` space prefix that the production sync path now requires.** Reverse-proxied Kibana ingresses (the standardised TIDE deployment shape post-4.1.14) route on `/s/<space>/...` and respond 404 / 302 to the bare path, which made Section 3 falsely report a connection failure on stacks where sync itself worked correctly — i.e. the diagnostic was lying. Changed the probe URL to `f"{url}/s/default/api/detection_engine/rules/_find?per_page=1"` so Section 3 mirrors the URL shape used by the live sync code path. The sibling per-`(siem,space)` probe in Section 4 was already correct and was not modified.

- **Stale rules in `detection_rules` for `(siem_id, space)` pairs no longer mapped in `client_siem_map` were never deleted (`app/services/sync.py:run_elastic_sync`).** Customer-reported symptom: DuckDB query showed 14 rules in a tenant DB while Kibana (and the UI Rule Health card after the orphan-prune in this same release) only knew about 7 — the other 7 were ghost rows from a `(siem_id, space)` pair the operator had since removed from the SIEM mapping. Root cause: the per-pair mirror reconcile (`reconcile_rules_for_siem_space` / `delete_rules_for_spaces`) only iterates the pairs CURRENTLY in `client_scope` (i.e. `client_siem_map` for the active tenant). When a mapping is removed (or repointed at a different space), the rows previously written under the OLD `(siem_id, space)` pair are never visited again — no fetch targets them, so neither `save_audit_results` (which only DELETEs rows in `synced_scopes`) nor the per-pair reconcile ever revisits them. They lingered indefinitely. Added an "Orphan-mapping sweep" pass at the end of `run_elastic_sync`, INSIDE the tenant context already pinned at the top of the function, that does `SELECT DISTINCT siem_id, space FROM detection_rules`, computes the set difference against `client_scope`, and `DELETE FROM detection_rules WHERE siem_id = ? AND space = ?` for each orphan pair. Logged with the SIEM label, count, and a reminder that baseline mappings to those rule_ids survive (no FK cascade — see Cascade-protection note below). Wrapped in its own try/except so a sweep failure can never break the sync's return value or surface as a `Synced 0` in the UI.

- **Cascade-protection audit (no code change required) — confirmed `step_detections.rule_ref` and `applied_detections.detection_id` / `host_id` / `system_id` are loose `VARCHAR` columns with NO foreign-key constraints to `detection_rules` (`app/services/tenant_manager.py:_init_tenant_db_schema`).** Operator concern was that the orphan-sweep above (or an upstream Kibana delete the existing reconcile path catches) would cascade and destroy hours of baseline-mapping work. Verified end-to-end: `step_detections (id PRIMARY KEY, step_id VARCHAR NOT NULL, rule_ref VARCHAR DEFAULT '', note VARCHAR, source VARCHAR)` and `applied_detections (id PRIMARY KEY, detection_id VARCHAR NOT NULL, system_id VARCHAR, host_id VARCHAR, client_id VARCHAR, applied_at)` have no `FOREIGN KEY (...) REFERENCES detection_rules(...) ON DELETE CASCADE` clauses. Deleting from `detection_rules` therefore CANNOT remove baseline / applied-detection mapping rows — operators are protected against silent data loss when rules are temporarily moved to staging or removed by upstream Kibana edits. Documented this contract in the orphan-sweep comment in `app/services/sync.py` so any future schema change that adds an FK fails review.

- **Orphan rule_refs in baseline / vulnerability views rendered as bare UUIDs instead of telling the operator the link still exists but the rule is currently absent (`app/inventory_engine.py:_resolve_rule_ref`).** Templates already fall back to `det.rule_ref or det.note or 'Rule'`, so a missing rule rendered as a 36-char UUID — visually indistinguishable from a free-text rule reference like `"AWS-CloudTrail-Anomalous-Access"`. After the orphan-sweep above (or any temporary promote-to-staging move that removes the production-side row), the operator needs to know "this mapping is intact, but the rule isn't currently in the production set" — not "the mapping is broken". `_resolve_rule_ref` now distinguishes the two cases: when the rule_ref is UUID-shaped but `SELECT name FROM detection_rules WHERE rule_id = ?` returns no row in the active tenant DB, returns `f"⚠ Detached: {rule_ref[:8]}"` (warning glyph + short UUID for debugging) instead of the raw UUID. Free-text rule_refs are returned unchanged. No template changes needed — every existing call site in `inventory_engine.py`, `templates/partials/baseline_coverage.html`, `templates/report/baseline_report.html`, etc. picks the new label up automatically because they all already wrap the value in `_resolve_rule_ref(...)` before rendering.

- **`Test Rule` button returned HTTP 500 with `ModuleNotFoundError: No module named 'log'` after a uvicorn hot-reload (`app/elastic_helper.py` top-level imports).** `elastic_helper.py` line 14 used a bare `from log import log_debug, log_error, log_info` which only resolves when `/app/app` is on `sys.path`. The legacy sync entrypoint (`app/services/sync.py:run_elastic_sync`) explicitly does `sys.path.insert(0, app_dir)` before its `import elastic_helper`, but the package-style callers (`from app.elastic_helper import preview_detection_rule` in `app/api/rules.py:test_rule`, `from app.elastic_helper import promote_rule_to_production` in `app/api/promotion.py`, and the two `test_elastic_connection_full` imports in `app/api/management.py`) do not — they relied on the module already being cached from a prior sync. With dev hot-reload (`uvicorn --reload`), edits to `elastic_helper.py` (which we made several of in this same release for the URL/timeout fixes above) trigger a re-execution of the module body; if `Test Rule` is the next caller before the next sync runs, the bare `from log import` raises and the test endpoint surfaces a generic 500. Wrapped the import in a `try/except ModuleNotFoundError` that falls back to `from app.log import ...`, so the module loads successfully under both call paths regardless of `sys.path` state. No behaviour change for the legacy sync path; package callers (`Test Rule`, promotion, management `Test Connection`) now survive hot-reload. Caught while auditing other dependencies of the 4.1.14 edits — the rule_logger and `_prune_orphan_scopes` work above did not introduce this fragility, but the simultaneous `elastic_helper.py` URL/timeout edits exposed it.

- **Rule Health metrics card showed orphan space buckets (e.g. `One: 7`) for spaces that had been removed from the SIEM mapping but whose rules still lingered in `detection_rules` until the next sync swept them (`app/api/rules.py:_prune_orphan_scopes`, wired from `app/api/rules.py:get_metrics` and `app/main.py:rule_health_page`).** When the operator changes a SIEM's mapped space (or removes a SIEM/space pair entirely from `client_siem_map`), the rows in the tenant's `detection_rules` for the old `(siem_id, space)` pair stay until the next successful sync runs the per-pair `DELETE FROM detection_rules` cleanup. In the meantime the metrics card kept showing the now-orphan space alongside the active ones, which made it look like the mapping change hadn't taken effect. Added `_prune_orphan_scopes(metrics, db, client_id)` which strips any key from `metrics.rules_by_space` and `metrics.rules_by_scope` whose space (resp. `siem_id|space` pair) is not in `db.get_client_siems(client_id)` for the active tenant. Called from both the page render (`/rules`) and the HTMX metrics swap (`/api/rules/metrics`) so the card and its background refresh agree. Pure UI filter — no rows are deleted; the sync path remains the source of truth for actual cleanup.

- **Scheduled rule-log export crashed every minute with `_duckdb.CatalogException: Catalog Error: Table with name detection_rules does not exist!` (`app/services/rule_logger.py:export_rule_logs`, `app/services/database.py:get_all_rules_for_export`, `app/scripts/diag_sync.py`).** Migration 45 (4.1.13) moved `detection_rules` from the shared DB to per-tenant DBs, but the APScheduler `rule_log_export_<siem_id>` jobs still called `db.get_all_rules_for_export(...)` from a background thread with no tenant context set, so `self.get_connection()` routed to the shared DB where the table no longer exists. Fix in three places: (a) `export_rule_logs` now iterates `db.list_clients()`, calls `set_tenant_context(resolve_tenant_db_path(cid, data_dir))` per tenant, fetches the SIEM/space-scoped slice from each tenant DB, deduplicates by `rule_id` (the same rule can legitimately appear in multiple tenants that mapped the same `(siem_id, space)` pair — first-write-wins is correct since the source rows are identical Kibana-side), restores the prior context after each tenant, and writes the deduped union to the per-`(siem, space)` log file. (b) `get_all_rules_for_export` now raises a clear `RuntimeError` ("requires an active tenant context… since 4.1.13 Migration 45 detection_rules lives only in tenant DBs") if called without a tenant context, so the next caller that forgets sees an actionable error in `tide.log` rather than a `Catalog Error`. (c) `diag_sync` section 5 stops blindly running `_check_dr_columns` against the shared DB — `detection_rules` is now expected to be ABSENT in shared, so the section prints a `+ shared DB: detection_rules absent (expected since 4.1.13 ...)` line, and only runs the column-diff (with an `X` marker) if the table is unexpectedly still there (which would mean Migration 45 didn't apply). The per-tenant column-diff in section 4 is unchanged.

- **Sync still failed with a generic `network error` banner after the URL fix below, because `fetch_detection_rules` masked the real exception (`app/elastic_helper.py:fetch_detection_rules`).** The `except (ConnectionError, Timeout)` handler did `last_err = str(e)[:120]`, which truncated the actual `ReadTimeout` / `RemoteDisconnected` / `ProtocolError` class+message into a 120-char snippet that surfaced upstream as the same generic "network error" the URL bug used to produce — so once the URL was right, the next failure mode was invisible. Three changes: (a) capture the full exception class and message — `last_err = f"{type(e).__name__}: {str(e)}"` — and emit a `log_debug` line with the full `traceback.format_exc()` per failed attempt so the operator sees `ReadTimeout` distinctly from `ConnectTimeout` / `RemoteDisconnected` / `ProtocolError`. (b) Lowered `PAGE_SIZE` from 1000 (Kibana documented max) to 100 — slow / large-rule-set Kibanas were taking >30s to construct a 1000-rule page, which the per-page `timeout=30` then aborted as a `ReadTimeout`. (c) Raised the per-page `session.get(..., timeout=…)` from 30s to 60s as additional headroom. (d) Added `"Connection": "close"` to the request headers so each page uses a fresh TCP connection — defensive against reverse proxies / load balancers that silently drop idle keep-alive connections from the `requests.Session` pool between pages.

- **Sync returned `network error, sync drift space default fetch 0/0 rules` for any SIEM whose only space was `default`, even though `Test connection` reported the correct rule count (`app/elastic_helper.py:fetch_detection_rules`, `_space_api_prefix`, `get_data_view_indices`).** The test-connection helper (`test_elastic_connection_full`) built `/s/default/api/detection_engine/rules/_find` for every space including `default`, but the sync loop, the promotion/exception-list helpers, and the data-view resolver all special-cased `space.lower() == "default"` and stripped the `/s/default/` segment to `/api/detection_engine/rules/_find`. Vanilla Kibana accepts both shapes, but reverse-proxy ingresses (the common air-gapped Kibana topology) route on the `/s/<space>/` prefix and 404 the bare `/api/...` form, which the sync orchestrator surfaced as the "network error / 0/0 rules" banner. Aligned all three URL builders onto the same `f"{base_url}/s/{space}/api/..."` formula already used by `test_elastic_connection_full` so test and sync hit the literal same path. `_space_api_prefix` keeps an empty/None guard with `log_error` so a missing `client_siem_map.space` value fails loudly instead of silently emitting `/s//api/...`. `fetch_detection_rules` now emits a `log_debug` line per `(siem, space)` on page 1 of each sync (`[sync url] GET <endpoint> (space=<space>)`) so the constructed URL is permanently visible in `tide-app` logs and any future regression is caught from logs alone. Iteration over distinct `(siem_id, space)` pairs is unchanged — only the per-pair URL string shape is normalised; AGENTS.md §8.2/§8.3 contracts intact.

### Added

- **`diag_sync` section 9: Dry Run Sync URL Construction (`app/scripts/diag_sync.py`).** New section between the verdict block (8) and the live sync trace (renumbered to 10) that walks every `(client, siem, space)` row in `client_siem_map` joined to `siem_inventory` and prints the exact HTTP method, fully-qualified URL, and redacted headers (`Authorization: ApiKey len=N first=... last=...`, `kbn-xsrf`, `Content-Type`) the app would send to fetch detection rules — without making any network call. Uses the same URL formula as `fetch_detection_rules` so the output is the literal request line the operator would see in `tide-app` logs after the fix above. Tagged per row with `[client=...] [siem=...] [space=...] [role=...]` so the output can be pasted to support unambiguously. Prints `[OK]` if all URLs use the `/s/<space>/api/...` shape, or `[WARN]` if any line uses the legacy bare `/api/detection_engine/...` form (which would mean the 4.1.13 special-case has crept back in).

- **Two Query Manager presets: "Rule count per SIEM/space/role" and "Top 10 rules for production vs staging" (`app/api/management.py:_QUERY_PRESETS`).** Both pinned to `target: "tenant"` because since 4.1.13 (Migration 45) `detection_rules` lives in each tenant DB, not the shared DB — running them in shared scope would error with `Table does not exist`. The first preset issues `SELECT m.siem_id, m.space, m.environment_role, COUNT(r.rule_id) AS rule_count FROM client_siem_map m LEFT JOIN detection_rules r ON m.siem_id = r.siem_id AND m.space = r.space GROUP BY m.siem_id, m.space, m.environment_role;` so the operator can see at a glance which `(siem, space, role)` triples are populated and which are empty (helpful for verifying the 4.1.14 URL fix actually delivered rules). The second issues `SELECT m.environment_role, m.siem_id, m.space, r.name FROM client_siem_map m JOIN detection_rules r ON m.siem_id = r.siem_id AND m.space = r.space ORDER BY m.environment_role, r.name LIMIT 10;` for a quick promotion-axis spot check. Both appear in the Built-in optgroup of the Query Manager preset picker.

## [4.1.13] - 2026-05-06

### Changed

- **Detection rules are now per-tenant; the shared `detection_rules` cache and the global background sync are gone (`app/services/database.py` Migration 45, `app/services/sync.py:run_elastic_sync`, `app/main.py` lifespan, `app/api/rules.py`, `app/api/promotion.py`, `app/api/sigma.py`, `app/api/management.py`).** 4.1.12 still kept a single `detection_rules` table in the shared `tide.duckdb` and tried to fan-out via `_distribute_rules_to_tenants` after every sync; the dedupe pass dropped any rule that legitimately existed in two `(siem_id, space)` pairs because it ran on the old `(rule_id, siem_id)` PK shape, leaving production with `0/0 rules` and a "network error / sync drift" banner. The cleanest fix per operator request was to stop sharing the table at all: each tenant's DuckDB now owns its own `detection_rules` rows, two clients holding the same rule is fine, and the next sync per tenant rewrites that tenant's copy. SCHEMA_VERSION bumped 44 → 45; Migration 45 drops the shared `detection_rules` table outright (cache-only, no operator state). `run_elastic_sync(client_id, force_mapping=False)` now requires `client_id` and writes directly to the tenant DB through `set_tenant_context()`; the previously-recursive `_distribute_rules_to_tenants` is stubbed to a deprecation warning. The `lifespan` startup task and the APScheduler interval job that called `scheduled_sync()` globally were both removed — sync now only runs on user actions: the `Sync` button on `/rules` or `/promotion`, `promote_rule` (now schedules a follow-up `asyncio.create_task(scheduled_sync(client_id=...))`), and `deploy_to_siem` (same). The `?scope=all` query parameter on `/api/sync/elastic`, `/api/rules/sync`, and `/api/promotion/sync` was removed; all three handlers take the active tenant from the existing `ActiveClient` dep. Page handlers that needed to read rules outside the API context (`client_detail` in `app/main.py`, `_render_client_siems_partial` in `app/api/management.py`, `sigma_page`) now resolve the tenant DB path with `resolve_tenant_db_path(client_id, settings.data_dir)` or move `_activate_page_tenant` ahead of the read so `db.get_sigma_coverage_data()` lands in the tenant DB. The `migration37_resync_required` advisory at the shared scope returns `False` (no shared cache to be empty); per-tenant scope is unchanged. The startup `[auth-banner]` log line replaced its `shared detection_rules=N` field with `shared detection_rules=N/A (per-tenant since 4.1.13)`. `RuleFilters.allowed_scopes` is retained as an optional parameter on `get_rules`, `get_rule_health_metrics`, and `get_promotion_metrics` (kept for `test/test_siem_isolation.py` back-compat), but every production call site stops populating it from `get_client_siem_scopes(client_id)` — tenant context is the isolation boundary now, not a per-call WHERE clause. Smoke-tested post-restart: `/health`, `/sigma`, `/rules`, `/promotion`, `/management` all return 200 against `admin/TIDEadmin`; `Migration 45` ran cleanly (shared `detection_rules` table absent, schema version = 45). Locks AGENTS.md §8.2 g4 / §8.3 — composite isolation now enforced at the database file level, not the predicate level.

## [4.1.12] - 2026-05-06

### Fixed

- **Sigma deploy returned 400 `SIEM/space mismatch detected` for every target except the first when one SIEM was mapped to multiple spaces (`app/api/sigma.py:deploy_to_siem`).** `db.get_client_siems(client_id)` returns one row per `client_siem_map` entry, so a SIEM mapped to both `staging` and `production` shows up twice. The deploy handler matched on `siem_id` alone (`for s in client_siems: if s.get("id") == siem_id: target_siem = s; break`), so it always picked the first row, then compared the form's `space` against `target_siem["space"]` and 400'd whenever the operator selected the other space — i.e. deploy worked for production but failed for staging on the same SIEM (or vice-versa, depending on row order). Switched the lookup to the composite `(siem_id, space)` key (matching the same `(siem_id, space)` row contract that the rest of 4.1.12 stands on) and split the failure message: "linked but not for this space" vs "SIEM not linked at all" so the operator knows which side to fix. Both error branches now correctly return 400.

- **Sigma Convert page SIEM/space selector rendered blank on every page load (`app/templates/pages/sigma.html`).** Two compounding bugs: (a) the option-render loop tagged `selected` on *every* target whose `environment_role == 'production'`, so a tenant with multiple production-tagged SIEMs ended up with the LAST `selected` winning silently and no visible "default" pick; (b) the init-time JS unconditionally executed `_spaceSelect.value = localStorage.getItem('pref-sigma-default-space') || 'staging'` even when neither the saved value nor the literal string `'staging'` matched any rendered `<option value=...>` — under most browsers that resolves to "no selection", which displayed as a blank box. Operators then submitted the form with an empty `space`, the request 400'd, and they (correctly) reported the dropdown as broken. Fixed by: introducing a server-side default precedence (first production target → first staging → first available) and rendering a leading `<option value="" disabled selected>Select target SIEM / space…</option>` placeholder so an empty list also shows explanatory text instead of a blank; and gating the localStorage restore to only apply when the saved value still exists in the current option set, otherwise keep the server-rendered default. The hidden `deploy-siem-id` sync (which copies `data-siem-id` from the chosen `<option>`) is unchanged and now correctly stays empty until the operator picks a real target.

- **Threat Landscape coverage on `/threats` leaked TTP coverage from any SIEM that shared a Kibana space-name with the active client (`app/services/database.py:get_threat_landscape_metrics`).** This reader was missed by the earlier composite-scope migration: it still called `get_client_siem_spaces(client_id, "production")` (returns space names only) and filtered with `LOWER(space) IN (...)`, exactly the AGENTS.md §8.2 g4 anti-pattern. On the multi-SIEM repro (`SIEMY` and `SIEMAGAIN` both exposing space `production`/`one`), Detective Comics' "Coverage" % counted TTPs from rules in `SIEMAGAIN` even though Detective Comics had no mapping there. Migrated to `get_client_siem_scopes(client_id, "production")` + the production `_scope_predicate(scopes)` helper so the SQL becomes `WHERE enabled = 1 AND ((siem_id = ? AND LOWER(space) = ?) OR ...)` — matches the contract used by `get_covered_ttps_for_client` / `get_technique_rule_counts_for_client` (heatmap path) and `get_rules_for_client` (rule grid path). Tenants with zero composite scopes still short-circuit to an empty covered-TTP set so the landscape cards render as zeros instead of accidentally returning the global table.

- **`detection_rules` PK extended to `(rule_id, siem_id, space)` so a single SIEM can legitimately expose the same rule in more than one Kibana space (`app/services/database.py` Migration 44, `app/services/sync.py` `_ensure_tenant_detection_rules_schema`, `app/services/tenant_manager.py` `create_tenant_db`).** After the per-client sync narrowing landed, manually triggering a sync from Detective Comics or Marvel produced log lines like `Constraint Error: Duplicate key "rule_id: c1d147ae-..., siem_id: ed377cfc-..."` and left the tenant DBs with zero distributed rows; Rule Health and Promotion both went blank for those tenants. Root cause: SIEMY exposed the same rule in spaces `one` AND `two`, the operator had legitimately mapped Detective Comics to both spaces (staging vs production routing through one SIEM), but the shared-cache PK `(rule_id, siem_id)` collapsed the two rows the moment sync wrote them, and the tenant `INSERT INTO detection_rules` then exploded on the same PK as soon as `_distribute_rules_to_tenants` resolved that client to both spaces. SCHEMA_VERSION bumped 43 → 44; Migration 44 drops + recreates the shared `detection_rules` table with `space VARCHAR NOT NULL` and `PRIMARY KEY (rule_id, siem_id, space)`, logging the wiped row count and pointing the operator at Settings → Sync to repopulate (cache-only data, zero operator state lost). Tenant DBs get the same PK two ways: (a) brand-new tenant DBs use the bumped `CREATE TABLE` in `tenant_manager.create_tenant_db`; (b) existing tenant DBs are detected via `PRAGMA table_info(detection_rules)` in `_ensure_tenant_detection_rules_schema` and dropped/recreated whenever the PK column set is missing `space`, then repopulated by the same call's `_distribute_rules_to_tenants`. `get_existing_rule_keys` and `get_existing_rule_data` now key by the triple, and the sync lazy-mapping restore loop unpacks `(rule_id, siem_id, space)` (defaulting `space` from `space_id` or `'default'` to mirror the `space_id.fillna('default')` normalization at insert time). Verified post-restart: PK migration ran cleanly across the shared DB and all 3 tenants, distribution succeeded with Primary=9, DC=5, Marvel=5 rules respectively, and the constraint-error log line is gone. The composite isolation contract from the earlier 4.1.12 fix is unchanged — readers still filter on `(siem_id, space)` pairs from `client_siem_map`. Locks AGENTS.md §8.2 g4 / §8.3.

- **`/promotion` returned 500 (`NameError: name 'staging_spaces' is not defined`) after the composite-scope migration (`app/main.py:promotion_page`).** The handler still referenced flat `staging_spaces` / `production_spaces` lists that no longer existed once the upstream code switched to composite `staging_scopes` / `production_scopes`. Derived the flat lists from the composite tuples (`sorted({sp for _, sp in staging_scopes})`) so legacy template references and downstream calls stop crashing.

### Performance
- **Heatmap matrix + dashboard rollup TTL caches and CSS virtualisation (`app/services/ttl_cache.py`, `app/api/heatmap.py`, `app/main.py`, `app/static/css/heatmap.css`):** Plan §7 + §1.4 + §4.3 (`.github/plan.md`). The two heaviest read-only requests in the app — the heatmap matrix build (`GET /api/heatmap/matrix`) and the dashboard rollup (`GET /dashboard`) — both fan out into 5–15 DuckDB queries plus pandas aggregations and a validation-JSON read on every hit, even when the user is just toggling a checkbox. Profiling against a tenant with 24 actors / 19 detection rules / 3132 Sigma rules showed warm-process matrix builds in the 200–400 ms range and dashboard rollups in the 300–500 ms range, dominated by repeat work that hadn't changed since the previous request.
  - **`app/services/ttl_cache.py` (new):** tiny thread-safe TTL cache (~120 LOC), per-tenant keyed, FIFO overflow eviction, lazy expiry on read (no background sweep — keeps the §2 isolation contract simple because no thread needs tenant-context plumbing). Two module-level instances: `heatmap_matrix_cache` (30 s TTL, 64 entries) and `dashboard_cache` (60 s TTL, 16 entries). Tenant scoping is enforced by putting `client_id or "__global__"` first in every cache key, so a stale entry from one tenant can never serve another (and the super-admin cross-tenant view is its own keyspace). Exposes `get`, `set`, `get_or_compute`, `invalidate`, `invalidate_prefix`, `stats` so future write endpoints can drop the right tenant slice on a baseline / rule edit instead of waiting out the TTL.
  - **Heatmap matrix cache (`app/api/heatmap.py:get_heatmap_matrix`):** the `HeatmapData` dataclass produced by the matrix-build is cached under `(client_id, frozenset(actors), show_defense, frozenset(normalised source filter))`. Frozensets so that `?actors=A&actors=B` and `?actors=B&actors=A` hit the same cell. Source filter strings are lowercased + stripped before hashing so display-vs-DB casing variants converge. On a hit the template still renders fresh (so user-specific HTMX targets and OOB metric swaps are correct) but the database fan-out and matrix assembly are skipped entirely. Smoke test against a single-actor matrix: cold 0.217 s → warm 0.103 s (~52% latency reduction).
  - **Dashboard rollup cache (`app/main.py:dashboard_page`):** `db.get_dashboard_metrics(client_id=_cid)` (the heavy combined-query function in `services/database.py:3031` — rule health + promotion + threat metrics, all sharing a single connection but still loading validation data and running multiple `df()` materialisations) is wrapped in `dashboard_cache.get_or_compute(("dashboard-metrics", _cid or "__global__"), ...)`. 60 s TTL because the rollup is an aggregate of aggregates and surfaces aren't expected to be rule-edit-pulse-live. Smoke test against a tenant: cold 0.435 s → warm 0.320 s (~26% latency reduction; remaining cost is the inventory/CVE engine calls and template render which the cache deliberately does not cover because their downstream state is more volatile).
  - **CSS virtualisation (`app/static/css/heatmap.css`):** added `content-visibility: auto; contain-intrinsic-size: 600px 400px` to `.tactic-column` and `content-visibility: auto; contain-intrinsic-size: auto 32px` to `.actor-checkbox-item`. Lets the browser skip layout + paint for off-screen tactic columns (the matrix scrolls horizontally on narrow viewports) and for off-screen actor checkboxes inside the 250 px scroll area (1 k+ actor enterprises). `contain-intrinsic-size` is the height/width hint that prevents scrollbar jump as elements enter/leave the viewport. Plan §4.3 explicitly called out these two surfaces as the visible-paint hotspots.
  - **Inline styles → CSS classes (deferred):** the plan also flagged "RAG cells = 5 CSS classes instead of inline styles" — inspection of `templates/partials/heatmap_matrix.html` shows the cells already use `class="ttp-card {{ cell.css_class }}"` (no inline `style=""`); the `cell.css_class` is one of three semantic classes (`status-gap`, `status-covered`, `status-defense`) computed in `models/threats.py`. The 5-class RAG palette wasn't actually present in the codebase — current model is a binary covered/gap split with a defense-in-depth third state, which already maps to CSS classes. No regression to fix; flagging the mismatch between plan and current code so a future "true RAG breakdown" effort doesn't reintroduce inline styles.
  - **APScheduler / single-flight guards (deferred):** Plan §1.4 also called for moving CISA-KEV / OpenCTI warm-up / Sigma re-index into APScheduler in lifespan and adding asyncio single-flight guards. Sigma pre-load is already a single lifespan call (boot logs: `Sigma rules pre-loaded: 3132 rules cached`). KEV / OpenCTI sync remain trigger-driven via the trigger directory the entrypoint watches, which is functionally equivalent to scheduled work without adding a new dependency. Single-flight guards across worker processes would need a Redis-backed lock (current deployment is single-container so the TTL cache itself acts as a coalescing window for the duration of the TTL — a second concurrent request gets the cached entry the first request's render produced as soon as it lands). ### Added
- **Bounded DuckDB connection pool (`app/services/connection_pool.py`, `app/services/database.py`, `app/main.py`):** Plan §3 (`.github/plan.md`). 4.0.x called `duckdb.connect(db_path)` on every request inside `DatabaseService.get_connection()` and closed it at the end of the context manager. With the per-tenant DuckDB-per-file routing introduced in 4.0.10, every page load paid the cost of a file-open, WAL replay, catalog scan, and schema introspection — on the heatmap and dashboard pages this dominated the request budget (5 sequential `db.get_*` calls = 5 reconnects).
  - **Pool shape:** LRU keyed by absolute db_path with `MAX_TENANTS=8` slots × `MAX_PER_TENANT=2` connections (matches the user-approved 8×2 sizing in `.github/plan.md`). Tunable via `TIDE_POOL_MAX_TENANTS` / `TIDE_POOL_MAX_PER_TENANT` for benchmarking.
  - **DuckDB tuning:** each fresh connection is configured once with `PRAGMA threads=4` and `PRAGMA memory_limit='512MB'` (overridable via `TIDE_DUCKDB_THREADS` / `TIDE_DUCKDB_MEMORY_LIMIT`). DuckDB's per-connection planner cache is now reused across requests, so the second hit on the same query is materially cheaper.
  - **Concurrency model:** the pool's per-slot `Queue(maxsize=MAX_PER_TENANT)` gates concurrent use of any single connection (DuckDB `DuckDBPyConnection` objects are not safe to share across threads). The old `_conn_lock` global mutex in `DatabaseService` is no longer the serialisation point — different tenants now run in parallel.
  - **Poison-pill safety:** if the wrapped block raises, the connection is closed instead of recycled. DuckDB does not document recoverability from arbitrary exceptions; a poisoned handle silently breaking the next request would be the worst-possible outcome.
  - **LRU eviction:** when an 9th distinct tenant arrives the least-recently-used slot is drained (`close()` on every cached connection) and the new slot takes its place. Counter exposed as `evictions` so we can tell whether 8 is too small for the deployment.
  - **Observability:** `/health` now returns a `db_pool` block (hits / misses / evictions / per-path occupancy) so ops can watch the hit-rate without scraping logs. After an empty boot + ~14 idle `/health` polls the shared DB shows **14 hits / 1 miss / 0 evictions** — i.e. one cold connect followed by 14 reuses. Heatmap & dashboard pages will show the same shape per-tenant.
  - **Shutdown:** `lifespan` now calls `pool.close_all()` after the scheduler stops so file locks are released cleanly before uvicorn exits — the bind-mount on `./data:/app/data` would otherwise leak a `*.wal` file.
- **Read-only DuckDB connect call-sites converged to writable** (`app/services/tenant_manager.py:64`, `app/services/sync.py:84`, `app/api/external_sharing.py:173`, `app/main.py:1532`, `app/api/management.py:2392`): DuckDB refuses to open a file with a `read_only` flag that disagrees with an existing handle (`Connection Error: Can't open a connection to same database file with a different configuration than existing connections`). Once the pool started keeping a writable handle alive across requests, every previous `read_only=True` call became a guaranteed runtime error on the second invocation. All five callers were already issuing only `SELECT` statements, so writable is semantically equivalent.
